# src/lib

Shared, reusable library code exported globally via the `$lib` import alias. Organized by domain: components, icons, styles, stores, types, and data.

---

## Directory Overview

| Directory | Purpose | Docs |
|---|---|---|
| `components/` | Component system (Atomic Design: atoms → molecules → organisms → dedicated) | [CLAUDE.md](./components/CLAUDE.md) |
| `icons/` | Inline SVG icon library (~20 icons, mostly line style) | [CLAUDE.md](./icons/CLAUDE.md) |
| `scss/` | Global SCSS: reset, variables, theming, breakpoints, typography, markdown, code highlighting, animations | [CLAUDE.md](./scss/CLAUDE.md) |
| `stores/` | Svelte stores (currently: `theme` with persistence) | [CLAUDE.md](./stores/CLAUDE.md) |

| `data/` | Static site data (currently: `meta.ts` for SEO defaults) | [CLAUDE.md](./data/CLAUDE.md) |
| `index.ts` | Re-exports all public library symbols for convenience | — |

## Key Architectural Patterns

### Component Composition
- **Atomic Design** — atoms (Button, Card, Image, Logo, Tag) compose into molecules (BlogPostCard, CodeBlock, Socials, ThemeToggle) which compose into organisms (Header, Footer, Hero, Features, etc.)
- **Dedicated components** — page-specific components live in `components/dedicated/{page}/` to avoid polluting the generic atomic layers with one-off logic
- **Polymorphic components** — Button and Card render as `<a>` or `<button>`/`<article>` depending on props

### Styling
- **CSS-driven theming** — all color/theme switching happens via `data-theme` attribute on `<html>`, no JS class manipulation
- **Semantic color tokens** — `--color--{name}` (e.g., `--color--primary`, `--color--text`) generated by SCSS mixin
- **Responsive design** — use named breakpoint mixins (`@include for-phone-only`) instead of hand-written media queries
- **No Tailwind in components** — uses scoped SCSS instead (Tailwind only configured but not actively used)

### State Management
- **Minimal stores** — only the `theme` store for persistence and DOM coupling
- **Local state preferred** — Svelte reactive variables (`let` declarations) for component-local state
- **Props over context** — favor prop drilling for clarity; context only when truly needed

### Types
- **Centralized in `utils/types.ts`** — all shared types live here to avoid circular imports
- **Type re-exports in `index.ts`** — consumers import types from `$lib` directly

## Maintenance

All subdirectories maintain their own `CLAUDE.md` file. When adding/removing/changing code in a subdirectory:

1. **Update the relevant `CLAUDE.md`** — document all public exports, component props, store contracts, etc.
2. **If the change affects library overview** — update this file (e.g., a new subdirectory, a major refactor to the component system)
3. **If the change affects project architecture** — update the root [../../CLAUDE.md](../../CLAUDE.md)

**Keep `index.ts` in sync** — re-export new public symbols so they're available via `$lib` shorthand.

## Common Import Patterns

```ts
// Components (from $lib/components)
import Button from '$lib/components/atoms/Button.svelte'
import Card from '$lib/components/molecules/Card.svelte'
import Header from '$lib/components/organisms/Header.svelte'
import Capabilities from '$lib/components/dedicated/papersy/Capabilities.svelte'

// Icons (from $lib/icons)
import ChatIcon from '$lib/icons/chat.svelte'
import GitHubIcon from '$lib/icons/socials/github.svelte'

// Stores (from $lib/stores)
import { theme } from '$lib/stores/theme'

// Types (from $lib/utils)
import type { BlogPost, Feature, TagType } from '$lib/utils/types'

// Data (from $lib/data)
import { siteBaseUrl, title, description } from '$lib/data/meta'

// Utility (from $lib/utils)
import { HttpRegex } from '$lib/utils/regex'
```
